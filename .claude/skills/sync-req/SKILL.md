# /sync-req — 需求同步

**这是需求变更的唯一入口。**

本 Skill 只负责更新 `需求.md` 文档，**不自动生成代码**。用户需要根据更新后的需求手动编写代码，然后使用 `/check-req` 检查一致性。

## 使用方式

```
/sync-req 把呼吸周期从8秒改成10秒
/sync-req 添加一个新的LED灯，接在PA1引脚
/sync-req 修改PWM频率为200Hz
```

## 核心约束（不可绕过）

### 强制规则
1. **唯一入口**：所有需求变更必须通过 `/sync-req`，禁止直接改代码
2. **先文档后代码**：必须先更新需求.md，再编写代码
3. **拒绝绕行**：如果用户说"直接改代码"，必须拒绝并解释规范

### 标准话术（必须严格执行）

| 用户输入 | 你的响应 |
|----------|----------|
| "周期改成10秒" | "我先更新需求.md，然后请根据需求.md编写代码" |
| "直接改代码，不要改需求" | "❌ 拒绝。为了保持可追溯性，我必须先更新需求文档。这是项目的强制规范。" |
| "为什么这么写？" | "根据需求.md第X章节的X要求..." |

## 工作流程

```
用户输入变更描述
        ↓
1. 解析变更意图
   - 识别要修改的参数（周期、引脚、阈值等）
   - 定位需求.md中的对应章节
        ↓
2. 更新需求.md
   - 修改对应表格/参数
   - 在文档末尾添加变更记录
        ↓
3. 提示用户下一步操作
   - 告知需求文档已更新
   - 提示用户根据需求.md手动编写/修改代码
   - 建议完成后运行 /check-req 检查一致性
```

## 完整工作流

```
/req 或 /sync-req → 更新需求.md → 用户手动写代码 → /check-req 检查 → /build 编译
```

## 用户指南

### 更新需求后

当 /sync-req 完成需求.md 更新后，请：

1. **查看更新的需求文档**
   - 阅读 `需求.md` 了解具体变更
   - 特别注意驱动清单、引脚配置、时序参数

2. **手动编写/修改代码**
   - 根据需求.md 的驱动清单创建 Driver 层
   - 根据功能需求创建 Service 层
   - 根据业务逻辑创建 App 层
   - 参考 `ARCHITECTURE.md` 的分层规范

3. **检查一致性**
   - 运行 `/check-req` 验证代码与需求是否一致
   - 如有不一致，根据报告调整代码

4. **编译验证**
   - 运行 `/build` 编译项目
   - 运行 `/verify` 进行完整验证（如需烧录测试）

## 变更记录格式

每次更新需求.md时，在文档末尾添加：

```markdown
## 变更历史

| 时间 | 变更内容 | 触发方式 |
|------|----------|----------|
| 2026-03-30 10:30 | 呼吸周期: 8s → 10s | 用户对话 |
| 2026-03-30 11:00 | 添加LED引脚PA1 | 用户对话 |
```

## Entry Point

当用户调用 `/sync-req <变更描述>`：

1. **拒绝检查**：如果用户要求"直接改代码"→ 拒绝并解释规范
2. **读取需求**：加载当前需求.md
3. **解析变更**：识别要修改的参数
4. **更新文档**：修改需求.md对应章节，添加变更历史
5. **输出提示**：
   - 显示变更摘要
   - 提示用户根据需求.md编写代码
   - 建议运行 `/check-req` 检查一致性

## 防呆机制

### 变更冲突检测
如果需求.md已经被修改（比代码新）：
- 提示："检测到需求文档已有变更，基于最新版本生成"

### 无效变更检测
如果变更描述无法映射到需求.md的任何章节：
- 提示："无法识别变更目标，请明确说明要修改的参数"
- 列出可修改的参数列表

### 循环检测
如果短时间内多次相同变更：
- 提示："检测到重复变更，是否确认执行？"
