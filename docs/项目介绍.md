# 工作流工具包介绍

本仓库是一套嵌入式 AI 工作流工具包，不是某个具体板卡或 IDE 的工程模板。

它提供：

- Agent 接手规则
- 上下文事实文件
- 构建、烧录、验证 adapter 入口
- 报告目录约束
- git 与日志强制规则
- 可生成的工程结构快照

它不提供：

- 默认三层或四层架构目录
- 默认测试目录
- MCU/板卡配置
- CubeMX 生成代码
- Keil 工程
- Claude Code 私有目录

## 设计目标

用户拿到 workflow 文件夹后，应当只需要“配置自己的项目事实”，而不是先删除不相关的历史模板文件。

## 事实源

| 文件或目录 | 作用 |
|------------|------|
| `.workflow/project.yaml` | 项目名、工具链、构建、烧录、测试、布局配置 |
| `.context/engineering.yaml` | 架构、目录边界、AI 修改约束 |
| `.context/hardware.yaml` | MCU、板卡、外设、资源所有权 |
| `.context/version.yaml` | 工具链版本、生成代码边界、关键脚本 |
| `.context/runtime.yaml` | 最近一次 build/flash/verify 状态 |
| `.agents/rules/` | 通用 Agent 规则 |
| `.agents/skills/` | canonical Skills |
| `reports/` | 当前证据快照 |

## 架构声明

workflow 不绑定 `App/Service/Driver`。这些目录可以存在于某个具体项目中，但不是 workflow 的默认资产。

接入项目时，应在 `.context/engineering.yaml` 中声明：

- 架构目录
- 依赖方向
- 模块职责
- 资源所有权
- 生成代码和人工维护代码边界

## 测试接口

workflow 默认不创建 `Test/` 目录，但保留测试接口。项目可以把测试放在任意目录，只要在 `.workflow/project.yaml` 中声明：

- `layout.tests`
- `build.test_command`
- `verify.command`
- `verify.report_path`

## Adapter

`tools/workflow.py` 当前保留这些 adapter 能力：

- `gcc`
- `cmake`
- `keil`

默认 `toolchain.type: none`，表示还没有接入具体工程。

## 日志与报告

- 当前工具输出写 `reports/`
- 日常进展写 `PROJECT_LOG.md`
- workflow 结构演进写 `EVOLUTION.md`
- runtime 当前状态写 `.context/runtime.yaml`

报告默认覆盖，不用时间戳堆文件。
