# STM32F103 模板工程项目介绍

## 1. 工程定位

本工程不是一个以某个具体业务功能为核心的应用项目，而是一套面向 STM32 嵌入式开发的可复制模板工程。它的重点在于建立一套稳定的工程结构、开发流程、分层边界、自动化工具链和质量门禁，使后续每一个新项目都可以从同一套规范起步。

模板当前以 STM32F103C8T6 最小系统板为基础，保留了一个呼吸灯示例作为分层架构和验证流程的演示样例。这个示例不是工程的主体，真正的主体是围绕需求、架构、代码生成、审查、编译、烧录、验证和日志沉淀形成的完整工作流。

一句话概括：

```text
这是一个“需求驱动 + 四层架构 + Agent 自动化 + 可配置工具链”的 STM32 项目模板。
```

## 2. 工程要解决的问题

普通 STM32 项目在持续迭代后容易出现以下问题：

- 需求只存在于聊天记录或口头描述中，代码为什么这样写无法追溯
- CubeMX 生成代码、业务代码和驱动代码混在一起，后续迁移困难
- App 层直接操作 GPIO、TIM、UART，硬件细节向上泄漏
- 同一硬件资源被多个模块同时控制，出现职责重叠
- 新增文件后忘记加入 Keil 工程，导致本地能看到文件但编译系统找不到
- 代码写完后缺少固定审查流程，ISR 阻塞、死循环、时序错误等问题滞后暴露
- 编译、烧录、测试、报告输出没有统一入口
- 模板本身的结构演进和业务开发日志混在一起，长期维护成本升高

本工程围绕这些问题给出一套约束明确的答案：目录固定、依赖方向固定、需求入口固定、报告路径固定、日志职责固定、自动化工具固定。

## 3. 核心设计思想

工程的设计思想可以拆成五条主线：

| 主线 | 说明 |
|------|------|
| 需求驱动 | `需求.md` 是唯一真相源，所有代码生成和验证都要回到需求文档 |
| 分层架构 | App、Service、Driver、HAL 各司其职，依赖只能向下 |
| 工具自动化 | 通过 Agent 工作流和 `tools/` 脚本串联需求、架构、审查、构建、烧录、验证 |
| 质量门禁 | 代码完成后必须进入审查和验证流程，严重问题修复后才能继续 |
| 工程可复制 | 保留模板骨架、规范文档、Agent 工作流适配器和脚本，清空业务内容即可启动新项目 |

## 4. 需求驱动工作流

本工程强制采用需求驱动开发。核心规则是：

```text
用户想法 → 写入需求.md → 基于需求.md 设计架构 → 基于需求.md 编写代码 → 验证代码是否满足需求.md
```

`需求.md` 的作用不是普通说明文档，而是后续开发、审查和验证的输入源。对话里的临时描述不能替代它。

相关 Agent 工作流分工如下。当前仓库提供 `.agents/skills/` 作为通用 Skill 源，`.claude/skills/` 只是 Claude/Codex 兼容镜像，其他 Agent 可直接遵循同一流程和工具命令。

| 工作流 | 输入 | 输出 | 职责 |
|------|------|------|------|
| `/req` | 用户原始需求 | `需求.md` | 将口头需求整理成标准需求文档 |
| `/arch` | `需求.md` | 架构设计文档 | 基于需求规划模块、接口、依赖和实现路径 |
| `/sync-req` | 需求变更 | 更新后的 `需求.md` | 处理需求变化，保持需求可追溯 |
| `/check-req` | `需求.md` + 代码 | 一致性报告 | 检查实现是否偏离需求 |

这套机制的目标是让每次修改都能回答三个问题：

- 这次改动来自哪个需求？
- 架构上应该落在哪一层？
- 如何验证它已经满足需求？

## 5. 四层架构

工程采用固定的四层结构：

```text
App → Service → Driver → HAL
```

其中 `App/`、`Service/`、`Driver/` 是本工程自己的稳定架构层，不随开发主机、IDE、编译工具链或 AI Agent 改变。平台变化应该被吸收到 `.workflow/project.yaml`、`tools/workflow.py`、`Core/`、`Drivers/` 和 `Driver/` 内部实现中，而不是改变这三层的职责边界。

各层职责如下：

| 层级 | 目录 | 职责 | 禁止事项 |
|------|------|------|----------|
| App | `App/` | 业务逻辑、算法、状态机 | 不能包含 HAL 头文件，不能直接操作硬件 |
| Service | `Service/` | 功能抽象、资源协调、对 App 提供稳定接口 | 不能直接操作寄存器，不能绕过资源所有权 |
| Driver | `Driver/` | 外设封装、寄存器或 HAL 调用、中断处理 | 不能依赖 Service 或 App |
| HAL | `Core/`、`Drivers/` | CubeMX 生成代码、HAL 库、CMSIS | 不承载业务分层代码 |

分层的重点不是“多几个目录”，而是建立长期可维护的依赖边界。App 层只关心业务状态，Service 层负责把硬件能力抽象成业务可用的服务，Driver 层隔离芯片和外设细节，HAL/平台层保持 CubeMX、厂商库、工具链和 MCU 差异的适配边界。

## 6. 目录结构规范

模板当前的核心目录如下：

```text
very_test/
├── App/                 # 应用层：业务逻辑、算法、状态机
├── Service/             # 服务层：硬件能力抽象与资源协调
├── Driver/              # 驱动层：GPIO/TIM/UART 等底层封装
├── Test/                # 持久化测试代码
├── Core/                # CubeMX 生成的平台代码
│   ├── Inc/
│   └── Src/
├── Drivers/             # STM32 HAL、CMSIS 等官方库
├── MDK-ARM/             # Keil 工程文件
├── tools/               # 自动化脚本
├── docs/                # 架构规范、工作流说明、参考知识库
├── reports/             # 审查与验证报告输出
├── AGENTS.md            # 通用 AI Agent 接手入口
├── .agents/             # 通用 AI 工作流资产和 canonical Skills
│   └── skills/
├── .claude/skills/      # Claude/Codex Skill 兼容镜像
├── CLAUDE.md            # Claude Code 兼容入口
├── .project_structure   # 生成的目录结构快照
├── 需求.md              # 当前项目需求
├── PROJECT_LOG.md       # 日常项目开发日志
├── EVOLUTION.md         # 模板工程演进日志
└── README.md            # 快速说明
```

强制规则：

- 新增业务模块只能进入 `App/`、`Service/`、`Driver/`、`Test/`
- `Core/` 只保留 CubeMX 生成的平台代码
- 禁止创建 `USER/`、`Middleware/`、`Application/` 等额外分层目录
- Keil Include Path 已围绕当前目录结构配置，移动目录会影响构建和 Skill 工具

目录边界可以按两类理解：

| 类别 | 目录/文件 | 是否随平台或工具链变化 |
|------|-----------|------------------------|
| 工程本身 | `App/`, `Service/`, `Driver/`, `Test/`, `docs/`, `.context/`, `.workflow/`, `.agents/`, `tools/`, `reports/` | 不随 Windows/Linux、Keil/GCC/CMake、OpenOCD/GDB 或 AI Agent 变化 |
| 平台/工具适配 | `Core/`, `Drivers/`, `MDK-ARM/`, `.vscode/`, `.claude/`, `very_test.ioc` | 可随 MCU、CubeMX、IDE、工具链或可选 Agent adapter 变化 |

因此，迁移平台时不是重新发明目录结构，而是保留工程本身目录，把差异吸收到 `.workflow/project.yaml`、`tools/workflow.py`、adapter、CubeMX 生成区和厂商库中。

## 7. 自动化工具链

工程中的 `tools/` 目录承担自动化执行职责，`AGENTS.md`、`.context/` 和 `.agents/skills/` 提供通用 AI 接手入口，`.claude/skills/` 则是 Claude/Codex 命令兼容镜像。其他 Agent 可以直接调用 `tools/`，也可以把 `.agents/skills/` 当作流程说明来读。

规则文件不以某个 Agent 私有目录为源头。根目录的 `AGENTS.md`、`CLAUDE.md` 只作为自动发现入口；通用规则放在 `.agents/rules/`，通用 Skills 放在 `.agents/skills/`。

| 工具或工作流 | 作用 |
|--------------|------|
| `/build`、`tools/build_keil.sh` | 编译 Keil 工程 |
| `/flash`、`tools/flash_keil.sh` | 烧录固件到开发板 |
| `/bf` | build + flash 的组合入口 |
| `/code-reviewer` | AI 语义代码审查，检查硬件配置、ISR 安全、死循环、资源冲突等 |
| `tools/code_reviewer.py` | 无 AI 环境下的静态审查兜底脚本 |
| `/driver-dev`、`tools/driver_dev.py` | 根据 datasheet 或参考资料生成 Driver 层代码并注册 Keil 工程 |
| `/verify` | 编译 TEST_MODE 固件，烧录后读取固件自验证 JSON 结果 |
| `tools/inject_test_mode.py` | 临时向 Keil 工程注入 `TEST_MODE` 宏 |
| `/dev`、`tools/dev_orchestrator.py` | 项目总控，读取状态、报告和日志，给出下一步动作 |
| `tools/git_guard.py` | 强制 Agent 在修改前检查 git 状态，并在最终交付前提交任务变更 |

这套工具链的重点是减少“靠记忆执行流程”。新增驱动、编译、烧录、审查、验证都应该优先通过固定入口完成。

Git 也是固定入口的一部分。任何 Agent 修改文件后，必须运行 git 状态检查、审查 diff、只暂存本次任务文件，并在最终交付前 commit；除非用户明确说不要提交。

## 8. 质量门禁

本工程把代码审查和验证作为开发流程的一部分，而不是完成后的附加动作。

标准流程：

```text
需求分析 → 架构规划 → 编写代码 → 自动审查 → 修复问题 → 编译验证 → 烧录测试
```

审查重点包括：

| 类型 | 检查内容 |
|------|----------|
| ISR 安全 | 中断中不能出现 `HAL_Delay`、`printf` 等阻塞操作 |
| 死循环保护 | 轮询等待必须有超时机制 |
| 需求符合 | GPIO、PWM、时序、通信行为是否符合 `需求.md` |
| 架构合规 | 是否存在反向依赖、跨层调用、重复初始化 |
| 资源所有权 | 同一 GPIO、TIM、UART 等资源是否被多个模块争用 |
| 栈与内存 | 大数组、共享变量、原子访问等风险 |
| 硬件配置 | 时钟、定时器频率、NVIC、DMA、UART 等配置是否合理 |

严重问题必须修复后才能进入下一阶段。这样做的目的是让工程在“能跑”的基础上尽量接近“可维护、可迁移、可验证”。

## 9. 固件自验证机制

模板中的 `/verify` 不只是简单烧录后观察现象，而是引入了固件自验证思路：

```text
编译 TEST_MODE 固件 → 烧录 → 固件自动运行测试 → 串口输出 JSON → 工具解析结果 → 生成报告
```

相关文件和机制：

| 文件或目录 | 作用 |
|------------|------|
| `App/test_framework.h` | 轻量 JSON 测试输出框架 |
| `App/test_breathe.h` | 示例测试用例 |
| `tools/inject_test_mode.py` | 自动注入和恢复 `TEST_MODE` 宏 |
| `.agents/skills/verify/verify.py` | 通用 Skill 源下的编译、烧录、读取串口、解析测试结果入口 |
| `reports/verify_report.md` | 验证报告输出 |

这个机制的意义在于，后续新增模块时可以逐步把关键验证逻辑沉淀为固件内自测，而不是每次都依赖人工观察。

## 10. 报告与日志体系

工程把“日常开发记录”和“模板结构演进”分开管理。

| 文件或目录 | 记录内容 |
|------------|----------|
| `PROJECT_LOG.md` | 日常业务开发任务、需求变更、问题修复、验证结果 |
| `EVOLUTION.md` | 模板工程结构变化、Agent 工作流迭代、工具链调整、规范重写 |
| `reports/` | build、flash、check-req、code-reviewer、verify 等工具输出的当前证据快照 |

这种拆分可以避免两个层面的信息混在一起：

- 项目开发日志回答“这个业务功能是怎么做出来的”
- 工程演进日志回答“这个模板为什么变成现在这样”
- 报告目录保存自动化工具的当前运行结果，默认覆盖旧报告；历史结论写入日志，不靠堆报告文件保存

## 11. 参考知识库

`docs/reference/` 用来沉淀跨项目可复用的硬件知识，例如 GPIO、定时器、UART、LED、电平极性和平台迁移注意事项。

它的价值在于：

- 换 MCU 或换外设时，不必从头翻 datasheet
- AI 生成代码时有稳定的硬件背景资料
- 团队可以把已验证的参数、时序和坑点持续沉淀下来
- Driver 开发可以从“临时查资料”转为“复用项目知识库”

## 12. 复制新项目的方式

使用本模板启动新项目时，建议保留工程骨架，清空业务内容。

应保留：

- `AGENTS.md`
- `CLAUDE.md`（Claude Code 兼容入口）
- `.project_structure`
- `.agents/`
- `.claude/skills/`（可选兼容镜像，使用 Claude/Codex Skill 适配器时保留）
- `tools/`
- `docs/`
- `reports/` 目录
- Keil 工程和 CubeMX 工程基础配置

通常需要清空或重写：

- `需求.md`
- `PROJECT_LOG.md`
- `EVOLUTION.md` 中与旧模板无关的记录
- `App/`、`Service/`、`Driver/`、`Test/` 中的示例业务代码
- README 中的新项目名称、芯片型号和目标功能说明

推荐启动流程：

```text
复制模板 → 修改工程标识 → 写入新需求 → 生成架构方案 → 分层实现 → 自动审查 → 编译烧录 → 验证记录
```

## 13. 当前示例的角色

工程中的呼吸灯代码只是模板自带的演示样例，它用于说明：

- App 层如何承载业务算法
- Service 层如何抽象 LED、PWM、日志能力
- Driver 层如何封装 GPIO、TIM、UART
- `需求.md` 如何描述功能和测试用例
- `/verify` 如何通过 TEST_MODE 执行固件自验证
- Keil 工程如何集成新增源码、头文件和宏定义

因此，评估本工程时不应只看“它实现了一个呼吸灯”，而应看它如何把一个小功能放进完整的工程规范中。

## 14. 项目价值

本模板的核心价值是把嵌入式开发中的隐性经验显式化：目录该怎么放、需求该从哪里来、代码该落在哪一层、硬件资源归谁、什么时候审查、怎么验证、报告放哪里、结构变化记在哪里，以及任何 AI Agent 接手时应该先读哪些事实。

对于后续项目来说，它提供的是一个可执行的开发秩序。功能可以替换，芯片可以迁移，外设可以扩展，但需求驱动、分层边界、自动化工具链和质量门禁这些骨架可以持续复用。
